# Apk Tool

Приложение-декомпилятор/компилятор Android-приложений для сборки в **AIDE**.

- Пакет: `com.apktool.app`
- Название: **Apk Tool**
- Формат проекта: AIDE (папки `src/`, `res/`, `assets/`, `AndroidManifest.xml`, `project.properties`)

## Как собрать

1. Скопируйте папку проекта на устройство.
2. Откройте её в AIDE (`Open an existing Android project`).
3. AIDE сам сгенерирует `R.java` и соберёт APK (Run).

`minSdk 21`, `target android-33` — совместимо со старыми и новыми Android.

## Возможности

Интерфейс:
- Кнопка **Выбрать APK** и под ней **apk → java** / **apk → smali**.
- Кнопка **Выбрать папку** (декомпилированный проект) и под ней **java → apk** / **smali → apk**.
- Ползунок адаптации сборки под Android 8–14 (0 = не менять оригинальный SDK).
- Окно логов с полосой прогресса в %, ошибки — красным.
- Кнопки **Копировать** и **Очистить** (очистка сбрасывает логи и выбранные файлы/папки).
- Всплывающие уведомления (Toast) и диалог результата при успехе/ошибке.
- Пофайловый вывод распакованных/добавленных файлов.

Пути вывода:
```
/storage/emulated/0/Download/ApkTool/decompiler/   (apk -> java/smali)
/storage/emulated/0/Download/ApkTool/compiler/     (java/smali -> apk)
```

## Движки (в `assets/`)

- `engine/apktool.jar` — Apktool 2.7.0 (apk↔smali, ресурсы). Уже дексован, грузится через `DexClassLoader`.
- `engine/jadx.jar` — JADX (apk→java), вызывается через публичный API (без `System.exit`).
- `engine/apksig.jar` + `key/testkey.*` — подпись APK (схемы v1+v2+v3).
- `bin/<abi>/aapt`, `aapt2`, `zipalign` — нативные бинарники под arm64-v8a, armeabi-v7a, x86, x86_64. Распаковываются под ABI устройства и делаются исполняемыми в рантайме.

## Что работает

- **apk → smali** — apktool `d`.
- **apk → java** — jadx API. Выдаёт ЧИСТЫЙ проект AIDE: `sources/` (.java) +
  `resources/` (AndroidManifest.xml, res/, assets/, lib/). **Готовый `classes.dex`
  и вложенный APK в проект НЕ кладутся** — код собирается заново при сборке.
- **smali → apk** — apktool `b --use-aapt2` + zipalign + подпись v1+v2+v3.
- **java → apk** — **настоящий конвейер AIDE** (компиляция из исходников):
  1. `aapt2 compile` + `aapt2 link --java` — ресурсы → `resources.arsc` **и**
     сгенерированный `R.java`;
  2. `ecj` — `sources/*.java` **+ свежий `R.java`** → `.class` (старый
     декомпилированный `R.java` исключается);
  3. `d8` — `.class` → `classes.dex`;
  4. упаковка `resources.arsc` + `classes.dex` + `lib/` + `assets/`;
  5. `zipalign` + подпись v1+v2+v3.

  Так как `R.java` и `resources.arsc` создаются в **одной** aapt2-сессии, ID
  ресурсов в коде и в arsc всегда совпадают — приложение запускается. Внешние
  зависимости (например Google Play Services) подключаются `.jar`-ами в папке
  `libs/` проекта (как в AIDE) — они попадают в classpath `ecj` и в `d8`.

## Изменения (v16) — размыкание бесконечного цикла-поиска (дефект jadx → чёрный экран)

**Это и есть настоящая причина чёрного экрана** (уточнение к v15: правка v15
корректна, но чёрный экран вызывал не `ReadScript`, а описанный ниже цикл).

**Симптом.** После `java → apk` камера запускается, Activity показана, камера
физически открывается, но экран **чёрный** без превью и интерфейса, приложение
**не падает** (нет FATAL). В логах устройства:

```
E/lgCamera: Photo sizes count=18                       ← камера открылась, параметры получены
E/InvokeHelper: Method getAutoExposure error ...       ← HUAWEI-метод getAutoExposure не найден рефлексией
(далее ~3 c тишины, потом только HiView-события — поток завис)
```

**Корневая причина (гарантированно-бесконечный цикл).** Метод
`com.android.camera.CameraView.getSupportedAutoExposures()` в декомпиляции jadx
содержит **мёртвый бесконечный цикл** — jadx потерял `break` в теле цикла-поиска:

```java
n = 0;
while (true) {
    if (n >= strArFields.length) {   // тело ПУСТОЕ — break потерян
    }
    n++;
}
```

Условие выхода (`n >= length`) есть, но `break` внутри `if` отсутствует, поэтому
`if` ничего не делает и цикл крутится **вечно**. Цепочка на HUAWEI MAR-LX1M:

1. `InvokeHelper.InvokeMethodFast(cParams,"getAutoExposure")` бросает исключение
   (метода нет на устройстве) → `String.valueOf(...)` даёт строку `"null"`;
2. в `InitCamera` срабатывает `if (m_strAutoExposure.equalsIgnoreCase("null") &&
   (strArExp2 = getSupportedAutoExposures()) != null ...)`;
3. `getSupportedAutoExposures()` входит в бесконечный цикл → **поток
   инициализации камеры виснет навсегда** → превью никогда не стартует →
   **бесконечный чёрный экран**.

Код компилировался и «запускался», но намертво вешал поток — классический
рантайм-дефект, невидимый на этапе сборки.

**Правка.** В `Engine.java` добавлен проход `fixDeadSearchLoop`, встроенный в
`autoFixJadxDefects`. Он находит **гарантированно-бесконечный** цикл вида
`while (true) { ... if (cond) { } ... }`, у которого:

- тело `if` пустое (только пробелы), и
- во всём теле цикла **нет ни одного** `break`/`return`/`throw`/`System.exit`
  (то есть выхода действительно нет),

и вставляет `break;` в пустое тело `if`, восстанавливая выход по уже
присутствующему условию — ровно то, что потерял jadx. Если тело `if` не пустое
или в цикле уже есть любой выход — цикл **не трогается** (без ложных срабатываний).

**Проверка (реальные ARM-инструменты aarch64, полный конвейер
`aapt2 → ecj → d8 → baksmali`).** По всем ~53 java-файлам проекта камеры проход
срабатывает **ровно 1 раз** — только на сломанном цикле в
`getSupportedAutoExposures`. В пересобранном `classes.dex` метод больше **не
содержит вечного цикла и корректно завершается** (`return-object`); ключевые
вызовы разбора параметров (`split/indexOf/substring/Vector/List.add`) в основной
ветке восстановлены. Ложных срабатываний на валидных `while (true)` (у которых
есть `break`) нет.

Логгер `ApkToolCrashLogger` (v14) **не тронут**.

## Изменения (v15) — восстановление чтения потока в цикле (дефект jadx → чёрный экран)

> **Уточнение (см. v16):** правка v15 верна и полезна (восстанавливает потерянное
> чтение потока), но именно чёрный экран камеры вызывал **не** этот дефект, а
> бесконечный цикл в `getSupportedAutoExposures` — см. раздел v16 выше. Формулировку
> «→ чёрный экран» ниже следует читать как «семантически мёртвый цикл чтения».

**Симптом.** Пересобранное через `java → apk` приложение камеры
(`rubberbigpepper` / `com.android.camera`) запускалось, но показывало **чёрный
экран** вместо превью. В логах устройства при этом:
`E/VideoReg: Script manufacturer HUAWEI, model MAR-LX1M` — приложение пыталось
прочитать `assets/videorecordscript.txt` (скрипт настройки камеры под конкретную
модель) через `rubberbigpepper.CommonCtrl.ScriptCompile.ReadScript`.

**Корневая причина (не манифест, не ресурсы — а сам код).** Сравнение байткода
метод-за-методом (baksmali оригинального `classes.dex` против нашего) показало,
что jadx при декомпиляции **теряет присваивание внутри условия цикла**. Оригинал:

```java
int nRead;
while ((nRead = cStream.read(byArBuffer)) > 0) { ... }
```

jadx выдаёт сломанный вариант без чтения потока:

```java
while (nRead > 0) { ... }   // nRead нигде не присваивается
```

Прошлая автопочинка (`declareMissingVariable`, см. v-историю) делала такой код
лишь **компилируемым**, объявляя `int nRead = 0;`. Результат — цикл `while (0 > 0)`
**никогда не выполняется**: скрипт настройки не читается, камера не
инициализируется под модель → чёрный экран. Код собирался и запускался, но был
семантически мёртвым.

**Правка.** В `Engine.java` добавлен новый проход `fixLostStreamReadLoop`,
встроенный в `autoFixJadxDefects` **до** `declareMissingVariable`. Он распознаёт
шаблон «условие цикла сравнивает счётчик, который выше по телу метода
присваивается из `stream.read(buffer)`, но в самом условии присваивание потеряно»
и восстанавливает каноничную форму `while ((nRead = cStream.read(byArBuffer)) > 0)`.
Дополнительно проход:

- удаляет прошлые заглушки-пустышки `int VAR = 0; // [ApkTool] дефект jadx:
  пропущенное объявление`, если они превратили цикл в мёртвый (обратная
  совместимость со старыми, уже испорченными проектами);
- сам объявляет счётчик (`declareReadCounters`) в правильной области видимости
  перед восстановленным циклом.

**Проверка (на реальных ARM-инструментах aarch64).** Полный конвейер
`aapt2 compile/link → ecj → d8 → baksmali` прогнан на исходнике камеры.
После правки байткод `ReadScript` **побайтово-эквивалентен оригиналу**: 12
вызовов = 12 вызовов (`InputStream.read`, `Charset.forName`, `ByteBuffer.wrap`,
`Charset.decode`, `StringBuffer.append`), цикл снова читает поток. Ложных
срабатываний на остальных 52 java-файлах проекта камеры нет — правка затронула
только `ScriptCompile.java` (1 восстановление).

Логгер `ApkToolCrashLogger` (v14) **не тронут**.

## Изменения (v14) — встроенный краш-логгер (диагностика «молчаливых» вылетов)

Пересобранное приложение может **молча не запускаться** (частая причина —
заглушки внешних SDK возвращают `null`, и оригинальный код падает с NPE ещё до
появления UI). Чтобы такие вылеты стало видно, ApkTool теперь **автоматически
встраивает краш-логгер** в любое собираемое приложение — и для `java → apk`, и
для `smali → apk`. Ручных действий не требуется.

**Что делает логгер.** В приложение добавляется класс-наследник
`android.app.Application` с именем `<package>.ApkToolCrashLogger`, который:

- в `onCreate()` ставит `Thread.setDefaultUncaughtExceptionHandler` — **любой
  необработанный вылет** (в любом потоке) пишется в лог со **стектрейсом**,
  после чего управление передаётся системному обработчику (штатный «крэш» не
  подавляется);
- пишет строку старта приложения;
- (java-путь, API ≥ 14) через `registerActivityLifecycleCallbacks` логирует
  создание/показ каждой Activity — видно, до какого экрана дошёл запуск.

**Куда пишутся логи.** В каталог данных приложения:
```
/storage/emulated/0/Android/data/<package>/files/apktool-log/app.log
```
Используется `getExternalFilesDir(null)` (не требует разрешений с Android 4.4+),
с fallback на явный путь `/storage/emulated/0/Android/data/<package>/files`.
Каждая строка — с меткой времени; файл дописывается (append).

**Как встраивается (не ломая приложение).**
- **Манифест.** В `<application>` прописывается `android:name` на наш класс.
  Если своего `Application` у приложения **не было** — просто ставится наш.
  Если **был** (`android:name=X`) — наш `ApkToolCrashLogger` **наследует X**,
  поэтому весь оригинальный код Application сохраняется, а мы лишь добавляем
  перехват после `super.onCreate()`. Повторная сборка логгер не дублирует.
- **java → apk.** Генерируется `ApkToolCrashLogger.java` в `sources/`, он
  компилируется вместе с приложением (`ecj → d8`) и попадает в `classes.dex`.
- **smali → apk.** Генерируется валидный `ApkToolCrashLogger.smali` +
  `ApkToolCrashLogger$Handler.smali` (обработчик вылетов вынесен в именованный
  класс — в байткоде анонимных нет) прямо в каталог `smali/`; `apktool b`
  ассемблирует их в `classes.dex`.

Проверено end-to-end на «camera lg»: класс логгера присутствует в собранном
`classes.dex` (и для java, и для smali), APK подписывается и верифицируется
(v1+v2+v3). Теперь причина «молчаливого» незапуска видна в
`app.log` (например `CRASH … NullPointerException` от заглушенного SDK).

## Изменения (v13) — сборка «camera lg» доведена до конца (проверено end-to-end)

Взят реальный пример из задачи: приложение камеры `com.android.camera`,
разобранное этим ApkTool (`apk → java`), которое **обратно не собиралось**.
Пройден весь конвейер `java → apk` на настоящих движках (нативный `aapt2`
arm64, `ecj`, `d8`, `apksig`) и устранены три причины отказа. Итог: проект
компилируется **автоматически** (0 ошибок ecj за 3 прохода автопочинки) и на
выходе — **подписанный APK** (проверено: `verified=true`, схемы v1+v2+v3).

- **Санация `values/public.xml` (`sanitizePublicXml`).** apktool сохраняет
  таблицу фиксированных ID (`<public .../>`), но в декомпилятах остаются
  «сироты» — записи для ресурсов, чьё определение apktool потерял (типично для
  Google Play Services: `cast_notification_id`, maps-enum'ы `none/hybrid`,
  wallet `buyButton` и др.). На них `aapt2 link` падает: «no definition for
  declared symbol». Так как `R.java` мы пересоздаём своей aapt2-сессией, ID
  фиксировать не нужно — сироты безопасно удаляются (на них никто не ссылается,
  иначе ecj позже укажет отсутствующее поле `R`). Собираем множество реально
  определённых ресурсов (`values/*`, `res/<type>/…`, `@+id/…`) и вычищаем
  лишние `<public>`. У камеры так убрано 34 сироты — только после этого `link`
  успешно генерирует `resources.arsc` + `R.java`.
- **Обёртка непойманных checked-исключений (`wrapChecked`).** ecj-ошибку
  «Unhandled exception type T» (jadx потерял `throws`/`try` у вызова, например
  `DocumentsContract.createDocument/deleteDocument` → `FileNotFoundException`)
  чиним, оборачивая оператор в `try { … } catch (java.lang.Exception e) { throw
  new RuntimeException(e); }`. Широкий `Exception` не требует import (ecj даёт
  лишь простое имя) и покрывает любое checked-исключение; проброс как unchecked
  сохраняет исходную семантику. Раньше покрывался только `InterruptedException`.
- **Объявление потерянной переменной (`declareMissingVariable`).** Ошибку
  «X cannot be resolved to a variable» (jadx «потерял» объявление, например
  счётчик `nRead` в условии цикла в методе, помеченном «Code decompiled
  incorrectly») чиним, вставляя объявление с дефолтом в начало тела метода. Тип
  по эвристике: сравнение/арифметика → `int`, иначе `Object`. Цель — дать коду
  скомпилироваться (как AIDE), а не восстановить утраченную логику.

Заглушки внешних классов (gms `AdView/AdSize/AdRequest`, apache
`NameValuePair/URLEncodedUtils`) идут только в classpath `ecj`/`d8` и в финальный
`dex` не попадают. Оставшиеся дефекты jadx у камеры (потерянная catch-переменная,
`wait()` без `throws`, метод без `return`, unreachable `break`, enum-как-числа)
чинятся уже существовавшими автопочинками — новые три закрыли остаток.

## Изменения (v12) — заглушки-View, apache.http, ещё дефекты jadx

Изучен декомпилированный AIDE (папка `AideMobile` в репозитории): он тоже
использует один `android.jar` в bootclasspath, требует `useLibrary
'org.apache.http.legacy'` для apache-классов и так же отвергает фатальные
JLS-ошибки. Значит остаточные ошибки — дефекты именно jadx-декомпиляции.
Добавлены их автопочинки, продолжающие итеративный цикл по выводу ecj:

- **`org.apache.http.*` теперь заглушается.** Начиная с Android 6 эти классы
  удалены из bootclasspath, в `android.jar` их нет — раньше ApkTool исключал их
  из заглушек и получал «import cannot be resolved». Теперь для отсутствующих
  apache-классов (`NameValuePair`, `URLEncodedUtils` и др.) генерируются
  заглушки, как для любой внешней библиотеки.
- **Заглушки-`View`.** Ошибка «addView(View,int) … is not applicable for the
  arguments (**AdView**, …)» означает, что заглушка передаётся туда, где нужен
  `android.view.View`. Такая заглушка перегенерируется как `extends
  android.view.View` с корректными конструкторами `super(Context)` (у `View`
  нет пустого конструктора).
- **`This method must return a result`.** Метод, который jadx не смог
  декомпилировать, получает дефолтный `return` нужного типа перед закрывающей
  скобкой — но только если тело ещё не завершается `throw`/`return` (иначе была
  бы «unreachable code»).
- **`local variable … may not have been initialized`.** Объявление переменной
  (jadx выносит его в начало метода) дополняется дефолтным значением по типу.
- **Безопасное комментирование unreachable.** Теперь комментируются только
  `break;`/`continue;`; `throw`/`return` не трогаются, чтобы не оставить метод
  без возврата (устранена регрессия, ломавшая методы-заглушки jadx).

## Изменения (v11) — итеративная автопочинка по ошибкам ecj

Компиляция теперь **дотягивается** до успеха за несколько проходов (до 8),
анализируя вывод ecj после каждого прогона — как это делает AIDE:

- **Обогащение заглушек членами.** Пустая заглушка внешнего класса (например
  gms `AdView`) получала ошибки вида «The method `setAdUnitId(String)` is
  undefined» и «`BANNER` cannot be resolved or is not a field». Теперь из этих
  ошибок автоматически дописываются недостающие **методы** (`public Object
  setAdUnitId(Object a0){return null;}`) и **поля/константы** (`public static
  Object BANNER = null;`) прямо в нужную заглушку, после чего компиляция
  повторяется. Так внешний API «доопределяется» ровно по факту использования.
- **Починка дефектов jadx по координатам ecj.** Ошибки, которые AIDE переживает,
  а строгий ecj отвергает:
  - «Unreachable catch block for T» → тип в `catch` расширяется до `Throwable`
    (Throwable всегда достижим — метод может бросить `Error`/`RuntimeException`);
  - «Unreachable code» → недостижимый оператор (`break;`/`return;`) целиком
    комментируется (баланс скобок сохраняется).

Правки берут номера файла и строки прямо из вывода ecj, поэтому точны и
локальны. Цикл повторяется, пока есть прогресс; при полной неудаче — честный
отчёт с файлами/строками для ручной правки, как в AIDE.

## Изменения (v10) — надёжная автопочинка дефектов jadx

Исправлены две частые ошибки компиляции декомпилята (были в camera lg):

- **`ex cannot be resolved to a variable`.** jadx выдаёт catch-блок вида
  `} catch (Exception e) { ex = e; ... ex.printStackTrace(); }`, где приёмник
  `ex` нигде не объявлен. Теперь первое присваивание `ident = param;` внутри
  такого catch превращается в объявление `Throwable ident = param;` — **локально
  и пометодно** (прошлая версия ошибочно смотрела на весь файл и пропускала фикс,
  если `ex` была объявлена в другом методе).
- **`unhandled exception InterruptedException`** на `X.wait()`, `X.join()`,
  `Thread.sleep(...)` в методах без `throws` (например `Thread.run()`). Такие
  вызовы автоматически оборачиваются в `try/catch (InterruptedException)` с
  восстановлением флага прерывания потока.

Обе правки точечные и безопасные — корректный код не затрагивается. Если после
них остаются иные дефекты jadx, ApkTool честно показывает файл/строку для ручной
правки (как AIDE).

## Изменения (v9) — автозаглушки внешних библиотек и автопочинка

Отвечает на вопрос «как добавить внешние библиотеки как .jar и автоматизировать»:
теперь это **делается автоматически** для любого разобранного apk.

**Автозаглушки (stub) внешних классов.** Перед компиляцией сканируются `import`-ы
во всех `.java`. Классы, которых нет ни в `android.jar`, ни в `sources/`, ни в
`libs/*.jar` (например `com.google.android.gms.ads.*`), автоматически заглушаются
сгенерированными `public class`-ами (с пустым/varargs-конструктором и вложенным
`Builder`). Их `.class` идут **только в classpath** компилятора и **не попадают
в финальный `dex`**. Если после первого прохода компилятор находит ещё
недостающие типы — заглушки **доращиваются итеративно** (до 4 проходов) из
текста ошибок ecj. Ручное добавление `.jar` больше не требуется для компиляции.

**Автопочинка типовых дефектов jadx.** Например `ex cannot be resolved to a
variable` (jadx «теряет» объявление catch-переменной): автоматически добавляется
объявление, код компилируется.

**Про запуск (честно).** Заглушки обеспечивают **компиляцию**. Если внешний
класс реально вызывается в рантайме (а не только импортируется), для работы
нужна его настоящая реализация — положите оригинальный `.jar` в `libs/` проекта
(он попадёт и в classpath, и в `dex`). Заглушки же не выполняют логику (пустые).
Часто это некритичный код (реклама, аналитика) — приложение стартует и работает,
а заглушённый путь просто ничего не делает.

**Остаточные дефекты jadx.** Некоторые ошибки — это баги декомпиляции в самом
коде (неверно восстановленный `try/catch`, checked-исключения у `wait()` и т.п.).
Их ApkTool честно показывает с именами файлов/строк — правьте вручную в
`sources/`, как и в AIDE (который тоже не соберёт код с ошибками компиляции).
100% автоматическая пересборка ЛЮБОГО декомпилята невозможна в принципе —
это ограничение декомпиляции, а не инструмента.

## Изменения (v8) — компилируемый вывод jadx

**Проблема.** ecj падал с `Syntax error on token "1", invalid ClassType` на
строках `new 1()`, `new 2()`. Это невалидный Java: jadx вынес анонимные классы
отдельно (`CameraView$1`) и не встроил их в файл — по умолчанию с настройками
«как есть». Такой код не компилируется, а прежняя сборка при этом всё равно
доводила дело до APK из частичных `.class` — он не запускался.

**Исправление.**
1. **jadx настроен на компилируемый код** при `apk → java`:
   `setInlineAnonymousClasses(true)` (встраивать анонимные классы —
   `new Runnable(){...}` вместо `new 1()`), `setInlineMethods(true)`,
   `setRenameValid(true)` + `setRenamePrintable(true)` (переименовывать
   невалидные идентификаторы), `setFallbackMode(false)`,
   `setShowInconsistentCode(true)`, `setDecompilationMode(AUTO)`.
2. **Сборка прерывается при ошибках компиляции** (как AIDE): APK из частичных
   `.class` больше не собирается. В логе — список файлов с ошибками (правьте их
   в `sources/`) и подсказка про `libs/*.jar` для внешних зависимостей.

Если проект был декомпилирован старой версией и содержит `new 1()` —
пере-декомпилируйте `apk → java` этой версией: код станет компилируемым.

## Изменения (v7) — настоящая логика AIDE вместо костыля

**Что было не так (v6).** Раньше инструмент клал в проект готовый `classes.dex`
и вложенный `.original.apk`, а при сборке просто клонировал оригинал, подменяя dex.
Это костыль, а не сборка из исходников: AIDE так не делает.

**Стало (v7).** `java → apk` теперь честно **компилирует исходный код**, как AIDE:
`aapt2 → R.java → ecj → d8 → упаковка → подпись`. Проект после `apk → java`
содержит только `sources/` и `resources/` — никаких готовых `.dex`/`.apk`.

**Почему это не ломает запуск (в отличие от прежней наивной пересборки).** Ключ —
`aapt2 link --java`: `R.java` генерируется **той же** сессией, что и
`resources.arsc`, поэтому целочисленные ID ресурсов в скомпилированном коде
гарантированно совпадают с ID в `resources.arsc`. Старый декомпилированный
`R.java` из `sources/` при этом исключается и заменяется свежим.

**Внешние зависимости.** Если декомпилированный код ссылается на классы, которых
нет в `sources/` (Google Play Services, сторонние SDK), положите их `.jar` в
папку `libs/` проекта — как в AIDE. Если библиотеки не хватает, `ecj` выдаст
понятную ошибку с именами недостающих классов (а не молчаливый краш).

## Изменения (v5)

**Декомпиляция apk → java выдаёт формат AIDE.** Результат:

```
<имя>_java/
├── sources/     ← classes.dex, разобранный в .java (jadx)
└── resources/   ← AndroidManifest.xml, res, assets, lib, META-INF
```

`classes.dex` **не** кладётся файлом в вывод — AIDE сам компилирует `.java` из
`sources/` обратно в `classes.dex` при сборке проекта (именно так этот формат и
собирается в AIDE). Ранее код ошибочно добавлял `classes.dex` в `resources/` —
это ломало формат AIDE; теперь dex-файлы из вывода удаляются.

Открывайте папку `<имя>_java` в AIDE как проект. Если приложение использует
внешние библиотеки (Google Play Services, Apache HTTP и т.п.), подключите
соответствующие `.jar` в AIDE — как при обычной сборке java-проекта.

## Про зависимости (честно)

`java → apk` компилирует исходный код из `sources/`. Внешние классы (SDK, которых
нет в исходниках) обрабатываются **автоматически**:

- **Для компиляции** — генерируются заглушки (см. «Изменения (v9)»), ручной
  `.jar` не нужен. Компилятор `ecj` использует `android.jar` + `libs/*.jar` +
  автозаглушки в classpath.
- **Для запуска** реально вызываемого внешнего кода — положите настоящий `.jar`
  библиотеки в `libs/` проекта: он попадёт и в classpath `ecj`, и в `dex` (через
  `d8`), т.е. заменит заглушку рабочей реализацией. Это и есть «добавить внешнюю
  библиотеку как .jar», как в AIDE.

Остаточные ошибки компиляции — это дефекты декомпиляции jadx в самом коде
приложения; ApkTool показывает файлы/строки, правьте их вручную в `sources/`.
